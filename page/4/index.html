JS的堆和栈

  • 堆（stack），存放基本类型，内存自动分配，自动释放
  • 栈（heap），存放引用类型，内存动态分配，不自动释放

浅拷贝和深拷贝

  • 浅拷贝：增加了一个指针指向已经存在的内存（会改变原函数）
  • 深拷贝：增加了一个指针并且申请一个新的内存（不会改变原函数）

1.流程

　　我们知道，一个IP地址对应多个域名，我们在地址栏输入域名，首先要获取此域名所对应的IP地址才可以，也就是——解析
　　解析域名，用到的是DNS，DNS通俗来说，就是域名的电话本，比如我们想给李四打电话（IP），我们又记不住李四的具体电话号码。这时候我们就需要一个电话本来翻翻（DNS），找到李四对应的电话，这样才能打过去（访问）
　　下面是具体的步骤：
　　1. DNS解析：从浏览器缓存–>系统缓存–>路由器缓存，然后一步步的进行递归搜索，搜索后给浏览器返回IP地址。
　　2. 建立连接（三次握手）

　　3. 客户端向服务器发送响应请求
　　4. 服务器处理请求，返回HTML页面
　　5. 发送AJAX
　　6. 断开连接（四次挥手）

2.http相关

1.方法

　　HTTP请求方式有OPTIONS、GET、HEAD、POST、PUT、DELETE、TRACE、CONNECT。通常我们用的只有GET、POST

2.Content-Type

　　Content-Type属性指定请求和响应的HTTP内容类型。如果未指定 ContentType，默认为text/html。
下面是几个常见的Content-type

1.text/html
2.text/plain
3.text/css
4.text/javascript
5.application/x-www-form-urlencoded
6.multipart/form-data
7.application/json
8.application/xml

附：http状态码：

状态码 意义
100 继续。客户端应继续其请求
101 切换协议。服务器根据客户端的请求切换协议。只能切换到更高级的协议
200 请求成功。一般用于GET与POST请求
204 无内容。服务器成功处理，但未返回内容
301 永久重定向
302 临时重定向
304 所请求的资源未修改，服务器返回此状态码时，不会返回任何资源
404 无法找到资源（网页）
500 服务器内部错误

说说JS的那些设计模式

发表于 2017-03-09

　　JS的设计模式很多，常用的有如下几个，有兴趣的还可以阅读极客学院的《学用JS设计模式》

传统的设计模式

　　1. 构造器模式
　　2. 模块化模式
　　3. 单例模式
　　4. 观察者模式
　　5. 原型模式
　　6. 外观模式
　　7. 工厂模式

　　我们今天重点说说JS的MV*模式以及模块化JS设计模式。

MV*模式

　　客户端根据用户的行为修改客户端Model–>客户端更新和该Model相关的View–>客户端Model发送sync请求到服务器，只包含改变了哪些数据–>服务器审核数据改动是否合法，只需回复是否修改成功–>客户端收到成功，什么都不用做；不成功，则把刚才改的View改回来，然后通知用户。（当然，也可以在客户端的Model里面也加入审核机制，这样对不合法数据的反应更快，但还是得保留服务器端的审核）

MVC模式

  • 模型（Model）：数据保存
  • 视图（View）：用户界面。
  • 控制器（Controller）：业务逻辑
    　　View（界面）触发事件–>Controller（业务）处理了业务，然后触发了数据更新–>不知道谁更新了Model的数据—>Model（带着数据）回到了View–>View更新数据

MVP模式

　　MVP中的P代表表示器。这是一个包含用于View的用户界面业务逻辑的组件
　　MVP定义了Presenter和View之间的接口，让一些可以根据已有的接口协议去各自分别独立开发，以此去解决界面需求变化频繁的问题，切断的View和Model的联系，让View只和Presenter（原Controller）交互，减少在需求变化中需要维护的对象的数量。

MVVM模式

　　MVVM中的VM代表是MVP的Presenter和MVC的Controller，而View和ViewModel间没有了MVP的界面接口，而是直接交互，用数据“绑定”的形式让数据更新的事件不需要开发人员手动去编写特殊用例，而是自动地双向同步。
　　比起MVP，MVVM不仅简化了业务与界面的依赖关系，还优化了数据频繁更新的解决方案，甚至可以说提供了一种有效的解决模式。

模块化

CMD、AMD

　　由于浏览器加载要看网速的快慢，所以必须使用异步加载的方式，这就是AMD产生的背景
　　它采用异步方式加载模块，模块的加载不影响它后面语句的运行。所有依赖这个模块的语句，都定义在一个回调函数中，等到加载完成之后，这个回调函数才会运行。
　　CMD和AMD的区别:CMD相当于按需加载，定义一个模块的时候不需要立即制定依赖模块，在需要的时候require就可以了，比较方便；而AMD则相反，定义模块的时候需要制定依赖模块，并以形参的方式引入factory中。

CommonJS

　　简单来讲，它是一个规范。
　　根据这个规范，每个文件就是一个模块，有自己的作用域。在一个文件里面定义的变量、函数、类，都是私有的，对其他文件不可见。

清除浮动

发表于 2017-02-26

　　说清除浮动，那必须谈谈为什么会出现这种情况，也就必须谈谈BFC，我们一个一个来

为什么会需要清除浮动？

　　浮动，从字面意义上，就好像一个气球一样漂浮在天空，所以就造成原来装气球的盒子（父元素、或者叫包裹容器）扁了（高度塌陷），所以我们要闭合浮动，也就是去掉高度塌陷。

清除浮动的6种方式

  1. 可以使用flex，但是IE支持性差
  2. 在元素下加一个div
    <div style = "clare:both">
  3. 将元素的容器也浮动（让箱子也飞到天上去，还是装气球，这样高度就回来了）
    <div style = "overflow：auto">

  4. 使用after这样的伪元素，给clearfix的标签加上，添加一个看不见的块元素来清除浮动。

  5. 触发BFC，只要触发BFC就能清除浮动。

  6. 建立一个匿名表格单元，其实也是触发BFC

浅谈BFC（块级格式化上下文）

　　其实说来说去，只要触发了BFC，就能够清除浮动，那么怎么去触发BFC呢？下面简单列举了几种触发方式，也就是说，有这些代码在，就可以做到清除浮动啦

  • float （除none以外）
  • overflow （除visible以外）
  • display （table-call、inline-block、table-caption）
  • position （fixed、absolute）

JS的垃圾收集

发表于 2017-01-20

　　今天来说说JS的垃圾回收吧
　　JS的垃圾回收有这么三种方式

　　1. 标记清除
　　2. 引用计数
　　3. 直接手动回收

标记清除

　　第一种是比较常用的，现代浏览器基本都会使用标记清除的方式来进行垃圾收集，大概分为两部分

  • 变量名进入环境——mark
  • 出了该环境——sweep

引用计数

　　而在老一点版本的IE上，我们还能看到另一种垃圾收集方式，就是引用计数的方式
　　这种办法大概意思就是，当给一个变量赋值，这个值的引用次数就+1，如果这个值又被赋给另一个变量，这个值的引用次数就-1，当值为0后，浏览器就认为可以对这个值进行垃圾回收了
　　但是引用计数有个bug，它存在一种循环引用的情况，你引用我我引用你，此消彼长，但是都不会减为0，这样就会导致该回收的垃圾一直得不到回收。。。
　　解决办法：
　　将变量赋值为null，手工断开JS对象与DOM之间的联系

直接手动回收

　　第三种就是手动回收咯
　　Window.CollectGarbage()

JS之原型链该怎么理解？

发表于 2017-01-13

　　我们把原型链和作用域放在一起说（总感觉这两个非常非常相似）

变量查找（作用域查找）

　　通过构造函数来创建一个对象

  1. 创建一个Object
  2. 准备构造函数的执行环境
  3. 将构造函数的this指向Object
  4. 执行构造函数
  5. 返回Object

　　在函数执行时，遇到一个变量，我们就会沿着它的作用域一步一步网上找，一直找到全局变量，那他的作用域就是全局。

属性查找（原型链查找）

　　首先我们要明确prototype和__proto__
　　每个函数都有一个原型对象属性：prototype，它包含两个成员属性：constructor&__proto__
　　每个对象都有一个隐藏的__proto__属性，它引用了创建该对象的函数的prototype，即：
　　foo.__proto__ === Foo.prototype //这里的__proto__成为”隐式原型”
　　原型链是当对象调用某一个属性的时候，首先要在自身的空间内找，找不到的话就去原型对象找，还找不到，就到object其原型对象去找，比如：
　　newTest.showName-->Test.showName-->Object.prototype._proto_//此为原型链终点，其值为null
　　具体使用两点：

  • 【读】属性会沿着原型链找
  • 【新增】属性不会去看原型链

　　几个公式：

  • foo.__proto__=== Function.prototype
  • obj.__proto__=== Object.prototype
  • Function.__proto__=== Function.prototype（Function也是一个函数，函数是一种对象，也有proto属性。既然是函数，那么它一定是被Function创建。所以——Function是被自身创建的。所以它的proto指向了自身的Prototype。）
